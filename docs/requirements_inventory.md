# Requirements Inventory

## Purpose

This document is the atomic requirement inventory for the full Yoked roadmap. It converts the active product, engineering, UX, and workflow files into implementation-plannable requirement IDs that can be traced to backlog tickets without relying on archive or reference-only material.

## Source-of-Truth Basis

Authoritative product, engineering, and UX inputs reviewed line-by-line:

1. `01-product-foundation/source_of_truth_manifest.md`
2. `01-product-foundation/yoked_prd.md`
3. `01-product-foundation/yoked_engineering_spec.md`
4. `02-ux-spec/yoked_ux_spec.md`

Workflow, repo-governance, and planning inputs reviewed line-by-line:

1. `AGENTS.md`
2. `README.md`
3. `CONTRIBUTING.md`
4. `.github/ISSUE_TEMPLATE/feature_ticket.yml`
5. `.github/ISSUE_TEMPLATE/bug_report.yml`
6. `.github/ISSUE_TEMPLATE/epic.yml`
7. `.github/ISSUE_TEMPLATE/human_action.yml`
8. `.github/PULL_REQUEST_TEMPLATE.md`
9. `.github/labels.yml`
10. `docs/codex/setup.md`

## Inventory Construction Rules

1. Every line of the four active source-of-truth files was reviewed before inventory creation.
2. Pure narrative context such as mission copy, user-problem framing, personas, and document-purpose text was reviewed but not assigned IDs unless it imposed implementation, workflow, UX, or release obligations.
3. Requirement IDs are source-scoped and stable:
   1. `MAN-xxx` for manifest requirements.
   2. `PRD-xxx` for PRD requirements.
   3. `ENG-xxx` for engineering-spec requirements.
   4. `UX-xxx` for UX-spec requirements.
   5. `AGT-xxx`, `RDM-xxx`, `CON-xxx`, `TPL-xxx`, `LBL-xxx`, `CDX-xxx` for repo workflow files.
4. Atomic means one requirement row may represent one coherent system contract, one feature contract, one screen contract, or one governance rule bundle, but it may not collapse unrelated systems.
5. Deferred capabilities explicitly present in the active files are inventoried and phase-tagged even when they are absent from launch UX.
6. Internal-only contracts are included when they create implementation work, data-schema work, API work, release work, or validation work.
